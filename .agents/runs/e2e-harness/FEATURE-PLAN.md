# Feature: End-to-end Test Harness skeleton

Slug: e2e-harness
Status: ready-for-flow

## Goal

A future test author can write `pnpm test:e2e` and watch a real Core spawn against a temporary Workspace, a real Web Client load in a headless browser, and a smoke test pass. The harness provides Playwright fixtures (`{core, workspace, page}`), a page-object scaffold for future product tests, and a documented seam for injecting a mock LLM provider into the Worker. **No product feature is being tested in this slice** — the deliverable is the framework itself plus enough hooks in Core and Worker to make it boot.

## Components touched

- [x] Core (`crates/core`) — print listening URL on stdout; honor a port flag (or `port=0`).
- [x] Worker (`packages/worker`) — provider seam; built-in mock provider gated by env var.
- [x] UI (`apps/web`, `packages/ui-sdk`) — none functional. A placeholder page must exist so the smoke test's "page loads" step has something to load.
- [x] Test Harness (`tests/**`) — new top-level package: Playwright + Vitest harness, fixtures, page-object scaffold, smoke test.

## Test-infra status

- Core: no `#[test]` items today. Bootstrap in slice 1 (Core's first slice).
- Worker: no `test` script today. Bootstrap in slice 4 (Worker's first slice).
- UI: no `test` script today. Not bootstrapped this feature; carve-out applies.
- Test Harness: doesn't exist today. The harness package itself bootstraps in slice 5.

## Slices

Ordered, stacked. Each slice has its own RED test that proves a single observable behavior end-to-end.

### Slice 1: Core prints `INKSTONE_LISTENING` URL on startup

- **Behavior proven:** When Core starts with `--port=0`, it binds an ephemeral HTTP port and prints exactly one line `INKSTONE_LISTENING http://127.0.0.1:<port>` to stdout before accepting connections. (If `--port=0` doesn't exist yet, this slice introduces it.)
- **Components:** Core
- **Contract delta:** No change.
- **RED test:** A Rust integration test in `crates/core/tests/listening_url.rs` that spawns the Core binary as a subprocess with `--port=0`, reads stdout line by line, and asserts the next line matches `^INKSTONE_LISTENING http://127\.0\.0\.1:\d+$`. The test sends SIGTERM and asserts clean exit.
- **Notes:** This slice also bootstraps Core's test infrastructure: a `tests/` directory under `crates/core`, dev-deps as needed (`assert_cmd`, `predicates`). The integration-test bootstrap is in service of this slice's RED test, not a separate slice.

### Slice 2: Core honors `--workspace=<path>` and exits cleanly on SIGTERM

- **Behavior proven:** Core opens a Workspace at the given path (creating it if needed), serves on a port, and exits cleanly on SIGTERM (no orphaned children, exit code 0 or 143).
- **Components:** Core
- **Contract delta:** No change.
- **RED test:** Extend `crates/core/tests/listening_url.rs` (or a sibling test) to: pass `--workspace=$tempdir`, assert the tempdir is populated with whatever Core writes on first open (at minimum, an empty/initialized Workspace marker), assert clean exit on SIGTERM.
- **Notes:** "What populates the Workspace on first open" depends on what Core does today. If today it does nothing, the test asserts the tempdir is reachable as a Workspace, not that specific files exist. The slice promises the *flag is honored*, not a Workspace schema.

### Slice 3: Placeholder web bundle is served at `/`

- **Behavior proven:** A request to `http://127.0.0.1:<port>/` returns HTTP 200 with a non-empty HTML body containing the string `Inkstone` (sufficient to prove the bundle was served, not a chrome 404).
- **Components:** UI (placeholder bundle), Core (static asset serving) — **two components, sequential**.
- **Order:** UI-first. Core's RED test asserts "GET / returns the Web bundle's HTML"; the bundle must exist on disk before the test can pass. UI is the producer, Core is the consumer.
- **Contract delta:** No change.
- **RED test:** Rust integration test that spawns Core, waits for the listening line, makes an HTTP GET to `/`, asserts 200 + body contains `Inkstone`.
- **Notes:** A real chat UI is out of scope. `apps/web/dist/index.html` may already exist from a `vite build`; if not, this slice ships a one-line placeholder `<!doctype html><title>Inkstone</title>` either committed under `apps/web/dist/` or generated by `pnpm -C apps/web build`. UI agent decides which based on the existing build setup. UI carve-out applies (no UI test infra). Core agent's portion (static-asset routing if not present) gets a Rust test.

### Slice 4: Worker exposes a provider interface

- **Behavior proven:** A TypeScript consumer can import `LLMProvider` from `@inkstone/worker`, implement it, and the type-check succeeds. The interface has at least one method: `complete(turn: TurnInput): AsyncIterable<TextDelta>`.
- **Components:** Worker
- **Contract delta:** No change. The interface is internal to Worker, not on the wire.
- **RED test:** A vitest test in `packages/worker/src/__tests__/provider.test.ts` that imports the type, defines a one-line conforming implementation, and asserts the type compiles. Bonus: assert that calling the implementation yields the expected stream shape.
- **Notes:** Bootstraps Worker's test infrastructure: adds `vitest`, a `test` script in `package.json`, and `__tests__/` convention. The interface signature here is a placeholder — slice 5 picks the first concrete shape the mock provider needs.

### Slice 5: Worker ships a mock provider that streams 3 chunks by default

- **Behavior proven:** Setting `INKSTONE_LLM_PROVIDER=mock` and `INKSTONE_LLM_FIXTURE=<path>` causes Worker to load the mock provider, which reads an ordered-list-of-Turn-responses fixture and streams each response as 3 `text_delta`-shaped chunks. Per-fixture override of chunk count works.
- **Components:** Worker
- **Contract delta:** No change.
- **RED test:** Vitest test that constructs a mock provider with a two-turn fixture (one turn with default chunking, one with `chunks: 1`), drives both turns through the provider's interface, and asserts the chunk counts and concatenated text.
- **Notes:** Worker is *not* spawned in this test. The mock-provider class is exercised in-process. Spawn-via-Core is out of scope for this feature.

### Slice 6: Test Harness package exists and ships a Playwright fixture that spawns Core

- **Behavior proven:** Running `pnpm -C tests/e2e test:e2e` (or equivalent) builds Core in debug mode, spawns it with `--workspace=$tempdir --port=0`, captures the listening URL from stdout, exits cleanly. **No browser yet** — this slice proves only the spawn primitive.
- **Components:** Test Harness
- **Contract delta:** No change.
- **RED test:** A vitest (or Playwright test runner) test in `tests/e2e/src/__tests__/spawn.test.ts` (or `.spec.ts`) that uses the harness's `spawnCore({ workspace, port: 0 })` helper and asserts:
  - The helper resolves with `{url: "http://127.0.0.1:<port>", child: <process>}` within a timeout.
  - The URL responds to a basic `GET /` (already proven in slice 3, just verifying again from harness).
  - On `await core.shutdown()`, the child process exits cleanly within a timeout.
- **Notes:** Bootstraps the `tests/e2e` package: `package.json` with `vitest` (or Playwright's runner), workspace integration, the `spawnCore` helper, the test. Adds the package to the root pnpm workspace. Adds a root `test:e2e` script that delegates to `pnpm -C tests/e2e test:e2e`. The harness's `spawnCore` performs `cargo build --manifest-path crates/core/Cargo.toml` as a pretest hook (one-time per run, cached).

### Slice 7: Test Harness Playwright fixture loads the page in a real browser

- **Behavior proven:** A Playwright test using the harness's `page` fixture navigates to the spawned Core's URL, the page loads (response status 200), and the body is non-empty. Tear-down closes the browser and Core cleanly.
- **Components:** Test Harness
- **Contract delta:** No change.
- **RED test:** A Playwright test in `tests/e2e/src/__tests__/smoke.spec.ts` that uses fixtures `{core, workspace, page}`, navigates `page` to `core.url`, asserts the response and body, lets the fixture tear down. The test must pass on a clean checkout via `pnpm test:e2e` from repo root.
- **Notes:** Adds Playwright + Chromium to `tests/e2e`. Introduces the Playwright fixture file that wires `core` (from slice 6) into Playwright's fixture system and adds the `page` fixture. This is the canonical example test the README points at.

### Slice 8: Page-object scaffold and README

- **Behavior proven:** A `ChatPage` class exists at `tests/e2e/src/page-objects/ChatPage.ts` with at least one method (`goto()`) and is used by the smoke test (replacing the inline navigation). The `tests/e2e/README.md` documents three sections: Harness primitives, Mock LLM, Page objects, with a pointer to the smoke test as canonical example.
- **Components:** Test Harness
- **Contract delta:** No change.
- **RED test:** Two assertions:
  1. The smoke test is rewritten to use `ChatPage.goto()` instead of `page.goto(core.url)`. The test still passes.
  2. A trivial vitest/import test asserts `ChatPage` is exported and `goto` is a function. (Lighter than a full Playwright run for the documentation/scaffold piece.)
- **Notes:** No new dependencies. Pure refactoring of slice 7's smoke test plus documentation. README is in scope; mock LLM section can reference slices 4–5 even though the harness doesn't yet wire them into Core (a future feature).

## Acceptance criteria (feature-level)

These hold once all slices have passed. Each is implicitly covered by one or more slice RED tests; this section makes it explicit.

- [ ] `pnpm install` succeeds at repo root with the new `tests/e2e` package added to the workspace.
- [ ] `pnpm test:e2e` (root script) builds Core in debug mode, runs the smoke test, exits 0.
- [ ] Smoke test spawns Core with `--workspace=$tempdir --port=0`, captures `INKSTONE_LISTENING <url>`, opens Chromium, navigates, asserts 200 + non-empty body containing `Inkstone`, tears down cleanly.
- [ ] Core exits cleanly on SIGTERM in the smoke test (no orphaned process).
- [ ] Worker exposes `LLMProvider` interface and a built-in mock provider conforming to it; mock is loaded by env var; mock streams 3 chunks by default with per-fixture override.
- [ ] Page-object scaffold (`ChatPage`) exists with `goto()` and is used by the smoke test.
- [ ] `pnpm check` still passes across the workspace.
- [ ] `tests/e2e/README.md` documents Harness primitives, Mock LLM, Page objects.

## Out of scope

- Any product-feature test. The smoke test only proves the harness can boot and load a page.
- WebSocket / JSON-RPC connection from test code. No `session/hello`, no protocol Requests.
- A real chat UI in `apps/web` capable of sending a message.
- Worker spawn during any test in this feature. The mock provider is exercised standalone (slice 5); Worker spawn-via-Core is a future feature.
- Cassette-based recording, programmatic fixtures, prompt-hash matching.
- CI integration (GitHub Actions etc.).
- Cross-Client harness reuse (TUI, capture).
- Security/auth on Core's listener. Loopback-only assumed.
- Performance benchmarks, snapshot testing, visual regression.

## Open questions

- **Existing Web bundle.** Slice 3 may need to ship a placeholder `apps/web/dist/index.html` if `vite build` doesn't already produce one. The UI agent for slice 3 discovers this at decompose time. If a real `vite build` is wired, prefer running it as a pretest hook in slice 6 rather than committing `dist/` artifacts.
- **`pnpm test:e2e` Rust dependency.** Slice 6 wires `cargo build` as a pretest hook. If the user's preferred convention differs (e.g. expects the user to have pre-built), the harness agent flags it.
