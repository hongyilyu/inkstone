- [x] Atomic writes + centralized dir mode for config + auth (H2/H3/H4 from May 2026 audit). Three related hazards on `~/.config/inkstone/`: (H2) `writeFileSync` is `open(O_TRUNC) + write` — a SIGKILL or disk-full mid-write truncates `config.json` / `auth.json`, next load sees invalid JSON, every preference / refresh token is silently lost on fallback to defaults; (H3) `writeFileSync({ mode: 0o600 })` only honors mode on creation, so overwriting a pre-existing 0644 `auth.json` leaves the refresh-token bytes on disk at 0644 for the duration of the write, then tightens via a follow-up `chmodSync` — a concurrent local reader can race-read during the window; (H4) `config.ts`'s `mkdirSync(CONFIG_DIR, { recursive: true })` uses default umask (0755), `auth.ts`'s uses `0700` — whichever writer creates the dir first wins, leaving `~/.config/inkstone/` world-listable if `saveConfig` runs before `saveKiroCreds`. Fix centralizes two primitives in `paths.ts`: `ensureConfigDir()` (creates `CONFIG_DIR` at 0700, always `chmodSync`s to tighten a pre-existing 0755) and `writeFileAtomic(dest, data, mode)` (writes to `${dest}.tmp` via `O_CREAT | O_EXCL | O_WRONLY` with the exact mode, `fsync`s, closes, `rename`s atomically). `O_EXCL` is deliberate: an orphaned `.tmp` from a prior crash fails loudly on the next save rather than silently clobbering, so the user sees the error in a toast instead of losing the file. `config.ts` `saveConfig` and `auth.ts` `save` both switch to `ensureConfigDir()` + `writeFileAtomic`; `auth.ts` drops the redundant trailing `chmodSync` (the tmp-file mode is now authoritative). `config.json` mode is tightened from 0644 to 0600 to match the "every file in this dir is sensitive" posture — no secrets today, but the dir already lives under 0700 so the tightening costs nothing and defends against future additions. New `test/persistence-atomic.test.ts` (8 cases): atomic-write mode on create, overwrite preserves mode with no 0644 window, orphaned `.tmp` fails loudly (O_EXCL), unbounded content size atomicity, `ensureConfigDir` idempotency, 0755 → 0700 re-tightening, `saveConfig` end-to-end produces 0600 file in 0700 dir. Scratch dir isolated from the preload's `CONFIG_DIR` so the vault pointer in `config.json` survives the test run; the `saveConfig` block re-seeds the preload's config after each test so downstream tests still see the vault. `bun run ci` green (140 tests).

- [ ] Move tool-arg summary format from the UI to the tool definition. The knowledge of "how do I format this tool's args for display" lives in the UI layer, which means (a) `ToolPart` is coupled to the tool vocabulary, and (b) a third-party agent can't supply its own format without editing Inkstone. Better shape: attach a `renderArgs?: (args) => string` function to the tool object itself, next to `name` / `description` / `parameters`. `ToolPart` looks up the tool by name and calls `tool.renderArgs?.(args)`, falling back to `⚙ <name>` alone when absent. Drops `tool-summary.ts` (~70 lines) entirely. Complication: pi-coding-agent's `readTool` / `writeTool` / `editTool` are factory outputs Inkstone doesn't own — we'd need either a cast-at-import intersection type (`ToolWithRenderArgs = AgentTool & { renderArgs?: ... }`) to attach the field in `backend/agent/tools.ts`, or an upstream PR to pi-coding-agent to standardize the hook. Inkstone-owned tools (`updateSidebarTool`) just grow a field. Deferred from PR #19's feedback — the current shape works and the refactor is architectural polish rather than a user-facing fix. Revisit when (a) a second agent lands with its own tools, or (b) the tool set grows beyond the four we have today.
