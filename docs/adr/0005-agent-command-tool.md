# 0005 — Three-tier model: agent, command, tool

Inkstone has three distinct concepts that must not collapse: an **agent** is a *context boundary* (a coherent system prompt + memory + skills + tool subset, isolated from every other agent so a Reader's article-workflow context can't bleed into a KB query and silently reshape it), a **command** is a user-typed verb at turn boundaries (`AgentCommand` declared per-agent on `AgentInfo.commands`, e.g. `/article foo.md`; or shell-level `CommandOption` for state-free verbs like `/clear`), and a **tool** is an LLM-invoked capability mid-turn (`read`, `edit`, `write` from the shared pool plus per-agent `extraTools`). The three answer different questions — "which context am I in?", "what does the user want done?", "what can the agent do to accomplish that?" — and collapsing any pair loses the ability to model one of those questions: a unified verb concept conflates user intent with LLM intent, and an agent that's "just its tools" loses both context isolation and the cache-stable system prompt that makes prompt caching pay off. The model is enforced by a system-prompt stability invariant — pi-agent-core reads `state.systemPrompt` once per `prompt()` call and Anthropic's `cache_control` plus Bedrock's `cachePoint` pin it byte-exact, so commands inject dynamic per-turn context via a `prompt(text)` callback that writes a user message rather than mutating the system prompt; reader's `/article` is the reference pattern.
