# 0008 — One agent per session's in-memory lifetime

A `Session` is bound to one agent for its in-memory lifetime; `selectAgent(name)` is legal only on an empty session (`store.messages.length === 0`) and throws otherwise, so once any of ADR 0006's commitment events fires (Tab pick, slash pick, first message) the agent is locked until `/clear`. The lock is load-bearing for prompt caching: pi-agent-core reads `state.systemPrompt` once per `prompt()` call and Anthropic's `cache_control` plus Bedrock's `cachePoint` pin the byte-exact system + tools prefix, so a mid-session swap would rewrite both and invalidate the cache for the rest of the conversation; bubble provenance (per-bubble `agentName` stamping) and permission clarity (zones are agent-scoped) are supporting reasons but cache stability is the hard wall. Persisted sessions replay into any agent context via the resume path — `clearSession()` → `selectAgent(target.agent)` → `restoreMessages(...)` in one `batch()` — which constructs a fresh in-memory lifetime bound to the stored agent and naturally satisfies the empty-session guard; mid-session swap-and-rebuild (OpenCode's pattern) was rejected because Inkstone's bubble-stamping and zone-overlay model don't have the bookkeeping OpenCode added to make swaps coherent. Cross-agent reference (e.g. user reads an article in Reader and wants to discuss it in KB) is supported via **fork**: a NEW session bound to the target agent with lineage captured at the persistence layer — never a swap on the existing session. Fork creates a fresh in-memory lifetime per the binding rule above; the exact mechanism (verbatim history inlining, summary, or reference + on-demand read tool) is a deferred design with flexibility preserved by the new-session-not-swap rule.
