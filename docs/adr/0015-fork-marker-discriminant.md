# 0015 — Forked-from marker uses its own `parts.type` discriminant

The display-only artifact a fork writes to its child session is a `parts` row of `type: "fork"` (widening the existing enum from `["text", "thinking", "file", "tool"]` to `["text", "thinking", "file", "tool", "fork"]`) on a synthetic assistant message in `messages`, with no corresponding entry in `agent_messages` — the child agent's LLM context is naive to it. Reusing `type: "tool"` with a magic name like `routed_from` was rejected because a fork artifact is *not* an LLM-invoked capability mid-turn (per ADR 0005's agent/command/tool boundary): no LLM emitted it, no permission rule gated it, no tool result came back. Naming it a tool would push category recovery into a stringly-typed `name` field on the discriminant, which is exactly the drift typed schemas exist to prevent.

The router's *own* `dispatch` tool call in the parent (router) session stays a real `type: "tool"` part — it really is an LLM-invoked tool call mid-turn. The asymmetry is now visible at the schema level: the parent has a tool-part (the LLM event that *caused* the fork), and the child has a fork-part (the artifact that *resulted from* it); two different discriminants for two categorically different things. Anything that walks `parts` looking for `type === "tool"` (renderers, exporters, future analytics, permission auditing, agent-message correlation) cannot accidentally sweep up fork markers — the type system makes the wrong-code-path leakage impossible rather than relying on naming convention.

The marker's payload today is `{ parentSessionId }` only — there's exactly one caller (the router per ADR 0007), so a caller-discriminating field on the payload would be inert. An `originator: "router" | "user"` field is the planned extension point: when user-initiated fork lands, renderers branch on it without name-magic on the discriminant (the discriminant says *what category of part it is* — a fork seam — and the payload says *how the fork was initiated*). Capturing this here so a future contributor doesn't reintroduce the field speculatively against ADR 0014's deferred shape.

The asymmetry between `messages` (carries the marker) and `agent_messages` (does not) inverts ADR 0013's existing pattern (where some `agent_messages` rows have no display counterpart) — that prior asymmetry already legitimized the split as the right seam for "display state without LLM-state counterpart" (and vice versa).
