# 0014 — Fork as the session primitive; router as one caller

Cross-agent reference is implemented as `forkSession({ parentId, targetAgent, seedMessages })` — a persistence-layer verb that creates a new child `Session` bound to `targetAgent` with `parent_session_id` set on the child and replays seed messages into the child under one `withTransaction` block (per ADR 0012). The router's freeform-message classification (ADR 0007) is the only caller today; a future user-initiated cross-agent reference (e.g. "I read this article in Reader, want to discuss it in KB") will use the same backend verb. Backend fork is uniform; per-caller UX divergence (the routing seam is seamless, user-initiated fork is a navigate-to-new-session affordance) lives entirely in the TUI.

The decision was forced by ADR 0008's "one agent per session's in-memory lifetime" rule, which forecloses mid-session swap-and-rebuild; cross-agent flows must therefore be new sessions, and a new session born from another session needs a primitive that captures the lineage. Naming the primitive after the *primitive* (fork) rather than after its first caller (route) keeps the schema honest — the discriminant column is `parent_session_id`, not `routed_from_session_id`, and the marker `parts.type` is `"fork"`, not `"routing"`. Caller-specific UX (the routing seam is seamless, user-initiated fork is a navigate-to-new-session affordance) lives in TUI render rules, not in separate persistence paths.

**Deferred:** an `originator: "router" | "user"` field on both the `forkSession` parameters and the marker payload. With one caller it would be inert — every value would be `"router"`. The field lands when the second caller (user-initiated fork) does, as a one-line schema widening. Capturing the deferral here so a future contributor doesn't reintroduce the field speculatively.

`createSession` stays a single-statement, agent-only constructor (no parent FK set; auto-committed by SQLite) for its existing call sites. `forkSession` is genuinely a different verb because it's a multi-write transaction — the child session row, the forked-from marker on the child, and any seeded `messages` / `agent_messages` rows must commit atomically or not at all (orphaned children with no marker, or markers without seeds, both break rendering). Allowing both shapes through one constructor would mean the "with parent" path silently widens to a transaction while the "without parent" path stays single-statement — exactly the dual-mode contract ADR 0012 was written to prevent.
