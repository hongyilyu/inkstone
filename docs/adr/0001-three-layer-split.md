# 0001 — Three-layer split: tui / bridge / backend

Inkstone splits `src/` into `backend/` (headless agent — pi-agent-core, providers, persistence), `bridge/` (the contract surface where independent things must agree on shapes — types like `DisplayMessage`/`AgentStoreState` plus the runtime that sits on those contracts), and `tui/` (Solid + OpenTUI), with imports flowing one direction: tui → bridge, tui → backend, backend → bridge. `bridge/` exists because two kinds of independent things need a place to agree: external upstreams (pi-agent-core ↔ OpenTUI, where each evolves on its own schedule) and Inkstone's own layers (backend ↔ TUI, with a web frontend on the roadmap making a second frontend consumer real rather than speculative). The rule for what lives here is "runtime that bridges independent things" — view-projection helpers like `tool-renderers.ts` (LLM tool-args → display strings, the openTUI ↔ pi-ai seam) and its `ansi.ts` sanitizer (defends the openTUI grid against pi-ai-surfaced escape codes), plus shape parsers like `frontmatter.ts` that both layers project from independently (vault-file format consumed by backend search/recommendations and TUI article rendering). Runtime that belongs to one layer stays in that layer — the event-reducer is inline in `tui/` because nothing else consumes it. Biome's `noRestrictedImports` requires `@tui/*`-style aliases for any cross-layer reference and also blocks relative-path equivalents (`../tui/...`), so the boundary can't be silently bypassed.
