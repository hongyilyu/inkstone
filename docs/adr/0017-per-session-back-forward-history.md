# 0017 — Per-session browser-style back/forward for the secondary page

The secondary page (the full-screen overlay that replaces the conversation when the user clicks a file chip or `@`-mention) has historically been a single global `Signal<State | null>` — "what is currently rendered" with no notion of previous or next. The user-reported gap was symmetric reverse navigation: pressing `Ctrl+[` to close a page, then `Ctrl+]` to step forward into the same page, mirroring browser back/forward. The implementation extends that gesture into a full per-session navigation graph: each session owns its own `{ back, current, forward }` triple, and switching sessions auto-displays whatever page that session was on (browser-tab semantics). State lives in `src/tui/context/secondary-page-history.ts` keyed by `sessionId`; `secondary-page.ts` keeps its public API (`getSecondaryPage`/`openSecondaryPage`/`closeSecondaryPage`) unchanged so existing callers (`file-part-handler.ts`, the sidebar back chip, `app.tsx`'s render gate, the `clear` action) work without modification.

**Browser semantics, not a single-slot undo.** The three operations follow the standard browser model: `navigateTo(page)` pushes current onto `back`, sets `current = page`, **clears `forward`**; `goBack()` pushes current onto `forward`, pops `back` into `current`; `goForward()` pushes current onto `back`, pops `forward` into `current`. The "clear forward on open" rule is what keeps the model coherent under mixed gestures — without it, opening page B after stepping back from page A would leave A reachable via `Ctrl+]`, which surprises users by time-travelling to a path they abandoned. Conversation is the implicit floor of `back` (represented as `current === null`, never pushed onto either stack); without that rule the stacks would accumulate phantom conversation entries every time a fresh page opens.

**Arrays, not single slots, for `back` / `forward`.** Today only one level deep is reachable (no page-from-page navigation gesture exists), so the arrays will have length 0 or 1 in practice. They are still arrays because the data shape shouldn't lie about its semantics — if a future feature ever lets a user click a link inside a markdown page that opens a second page, it slots into the existing model unchanged. The behavioral-guidelines warning about speculative infrastructure cuts the other way: changing the data structure later requires re-auditing every nav function for off-by-one errors, whereas the array form is the structurally correct shape from day one. The cost is two `State[]` instead of two `State | null`.

**Sibling module to `secondary-page.ts`, not in the agent store.** `secondary-page.ts` is a presentation container — its job is "what is currently rendered". Mixing session-scoping and history into it would invert the dependency: a presentation primitive ends up needing reactive access to `currentSessionId` from the agent store, which conflates layers. The sibling-module pattern matches `prompt-draft.ts` precedent (also session-scoped UI state, also kept out of the reducer). The agent store remains for backend-event-driven state (messages, sidebar sections, session metadata); per-session UI navigation is a separate concern with its own module-level signals wrapped in `createRoot` for the same reason `prompt-draft.ts` and `secondary-page.ts` are: process-lifetime ownership and test-harness isolation.

**Process-lifetime only.** Mirrors `prompt-draft.ts` — quitting Inkstone discards every session's nav graph. The reported gap is "navigate and come back," not "quit and come back," so disk persistence would be speculative infrastructure. If a real need surfaces it lands as an additive feature (write to `~/.local/state/inkstone/...` keyed by sessionId, mirror prompt-draft's hypothetical disk-stash). Within a process, switching session A → B → A preserves A's nav graph because the per-session `Map<sessionId, SessionNav>` is keyed by id and never wiped on resume — only on `/clear` (which destroys the session entirely, so its nav graph going with it is correct).

**The "close the page" verb has two intents that had to be split.** Pre-history, `closeSecondaryPage()` was called from five sites with two distinct meanings: navigation gestures (ESC/Ctrl+[ in the layout-keybinds hook, the sidebar back chip) and session resets (`clear.ts`, `resume.ts`). Conflating them under one function makes "wipe history on `/clear` but preserve it on `/resume`" structurally awkward — the function would have to take a flag, or every caller would have to coordinate with a separate history wipe. Split as `goBack()` (history-aware navigation) and `closeSecondaryPage()` (history-wiping reset, used only by `/clear` now). `resume.ts` no longer calls either — the bridge effect in `use-layout-keybinds.ts` watches `subscribeSessionId()` and forwards changes into `setActiveSession(sid)`, which atomically swaps the rendered page to the new session's `current`. That's the contract that makes browser-tab semantics work without explicit wiring at every resume site.

**Trade-off accepted.** `current` in `SessionNav` and the `view` signal in `secondary-page-history.ts` are redundant — `view` is a derived projection of `map.get(sid)?.current`. The duplication is intentional: it lets every existing caller of `getSecondaryPage()` keep working unchanged (a single signal read, no `Map` lookup at every render), at the cost of one extra `setView` call per nav operation. The alternative — making `getSecondaryPage()` a memo over `state()` — would force every reader through the `Map` lookup on every render and would not preserve the public API's shape. The two writes stay in lockstep because every nav function is the only writer to both signals; there's no path that mutates one without the other.

The session-id bridge in `use-layout-keybinds.ts` reads `subscribeSessionId()` inside a `createEffect` and calls `setActiveSession(sid)`. `setActiveSession` reads `state()` (to compare the previous sid) and writes `setState(...)`. Without `untrack` around the body, the effect tracks `state` as a dependency, so every nav write would re-fire the effect, which would call `setActiveSession` with the same sid, hit the early-return — and Solid's batched scheduler under load surfaces this as a stack overflow even though correctness is preserved. The `untrack` wrap inside `setActiveSession` breaks the cycle without changing observable behavior; mirrors the same `untrack`-on-snapshot-read pattern from `prompt-draft-bridge.tsx`.
