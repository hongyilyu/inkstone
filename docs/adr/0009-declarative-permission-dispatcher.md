# 0009 — Declarative permission dispatcher

Tool-call permission policy is declarative data, not procedural code: a single `beforeToolCall` hook delegates to `dispatchBeforeToolCall` in `permissions.ts`, which evaluates a tagged-union array of `Rule` objects (`insideDirs`, `confirmDirs`, `frontmatterOnlyInDirs` — sized to current needs plus one) on first-block-wins ordering. Rules come from two sources: per-tool **baselines** declared as a required `baseline: Rule[]` field on each `InkstoneTool` definition (today: `read`/`write`/`edit` all enforce `insideDirs: [VAULT_DIR]` — the hard vault boundary every agent must respect; `update_sidebar`/`search`/`list_keys`/`suggest_command` declare `[]` to make "no baseline rules" an explicit contract) and a per-agent **overlay** produced by `AgentInfo.getPermissions?()` that owns the agent's full directory-level workspace policy (`insideDirs` allowlist + `confirmDirs` confirmations + `frontmatterOnlyInDirs` shape restrictions). Read-only enforcement is implicit: paths an agent shouldn't write are simply absent from `insideDirs`, and the dispatcher rejects them with a generic "Path must be within …" reason. Baselines are fully data on the tool; the overlay *producer* stays a function so state-dependent values can be inlined fresh per call, but the rules it returns are pure data. Procedural alternatives were rejected because reader-specific knowledge would have to live in the shell to dispatch correctly — the declarative shape lets each agent own its own rules in its own folder without the shell knowing reader vocabulary, and Biome enforces that the shell can't reach into agent internals (cf. ADR 0003). Permission rules are also the single source of truth for the system prompt's `<your workspace>` block: `composeWorkspaceBlock` (in `compose.ts`) reads the same `composeOverlay(info)` output the dispatcher consumes and projects each rule kind into prose (`insideDirs` → "You can write to:" with a per-dir `(write freely)` / `(confirm before each write)` suffix derived from `confirmDirs` membership, `frontmatterOnlyInDirs` → "Edits restricted to frontmatter in:"). The LLM's stated workspace and the dispatcher's enforced workspace are literally the same bytes; drift is structurally impossible.
