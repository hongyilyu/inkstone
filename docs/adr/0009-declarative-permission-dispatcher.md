# 0009 — Declarative permission dispatcher

Tool-call permission policy is declarative data, not procedural code: a single `beforeToolCall` hook delegates to `dispatchBeforeToolCall` in `permissions.ts`, which evaluates a tagged-union array of `Rule` objects (`insideDirs`, `confirmDirs`, `blockInsideDirs`, `frontmatterOnlyInDirs` — sized to current needs plus one) on first-block-wins ordering. Rules come from two sources: per-tool **baselines** registered at module load (today: `read`/`write`/`edit` all enforce `insideDirs: [VAULT_DIR]` — the hard vault boundary every agent must respect) and a per-agent **overlay** produced by `AgentInfo.getPermissions?()` for agent-scoped additions; the *rules* are pure data but the *producer* is a function so state-dependent values can be inlined fresh per call. Procedural alternatives were rejected because reader-specific knowledge would have to live in the shell to dispatch correctly — the declarative shape lets each agent own its own rules in its own folder without the shell knowing reader vocabulary, and Biome enforces that the shell can't reach into agent internals (cf. ADR 0003). Permission rules are also the single source of truth for the system prompt's `<your workspace>` block: `composeSystemPrompt` projects the merged rules into prose (e.g. `insideDirs` → "you can write to these dirs", `confirmDirs` → "...with confirmation", `blockInsideDirs` → "...these are read-only", `frontmatterOnlyInDirs` → "...where edits must target frontmatter only") so the LLM's stated workspace and the dispatcher's enforced workspace cannot drift — they are literally the same bytes.
