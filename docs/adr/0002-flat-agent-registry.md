# 0002 — Flat agent registry, composition over inheritance

Agents are plain `AgentInfo` objects in a flat registry; shared behavior lives in a base layer (`BASE_TOOLS`, `BASE_PREAMBLE`, `composeTools`, `composeSystemPrompt`) applied at runtime by composer functions, not via a `BaseAgent` class. Inheritance was rejected because agents diverge along multiple axes — prompt, tools, permissions, persona, session state — that don't share a single inheritance spine, so a base class would be mostly syntactic and would fight composition rather than enable it; every modern agent framework surveyed (OpenCode, Claude Code, Cursor, Codex) reached the same conclusion. Keeping `AgentInfo` as data (not a class) also means every field is inspectable at a glance, the registry is a literal array, and there is no hidden instance state — the trade-off is losing the language-level "override this method" ergonomic, which composer functions handle adequately.
