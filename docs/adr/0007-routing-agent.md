# 0007 — Routing agent as first-message classifier

Freeform text typed on the open page (no leading slash, no prior Tab pick) goes to a **router** — a normal `AgentInfo` registry entry whose only job is to classify the user's first message into a target agent and call its single tool, `dispatch({ agent })`, whose `args.agent` is a provider-enforced enum over `AGENTS` minus the router itself. The router is **stateless** and **first-message-only**: `dispatch.execute()` calls `forkSession()` (per ADR 0014) and the TUI aborts the router's in-flight turn the moment the tool resolves, then runs the existing resume-flow into the child (`clearSession` → `selectAgent(target)` → `restoreMessages(...)` per ADR 0008); the router session is sealed and never takes a follow-up turn. Misroute correction is a *new* freeform open-page submit (which spawns a new router session), not a multi-turn dialog with the original router session.

Slash commands and explicit Tab picks bypass the router entirely because they ARE the classification — re-running an LLM call to re-derive the verb the user already typed would be silly. The router is on the agent registry (single model, composes with the existing `compose.ts` / permissions infrastructure) and is included in the Tab cycle alongside Reader/KB so the user can cycle back to "let the router decide" after Tabbing past it; Inkstone treats "an agent whose job is to dispatch" as a real agent in both the architectural and the navigation sense.

User-visible routing provenance lives in the *child* session as a forked-from marker (per ADR 0015), not in the router session — router sessions are backend infrastructure (their existence gives the child's `parent_session_id` FK a target and the `dispatch` tool-call somewhere to persist) and are filtered out of the sessions list so the user only ever navigates to child sessions. Each agent's `AgentInfo.description` doubles as the router's "when to pick this agent" prompt, with one tiebreaker rule baked into the router's system prompt — *"if unsure between two, prefer the freeform-capable agent"* — so ambiguous classifications land on agents with a plain-chat mode rather than agents that force a workflow.
