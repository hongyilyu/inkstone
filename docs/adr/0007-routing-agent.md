# 0007 — Routing agent as first-message classifier

Freeform text typed on the open page (no leading slash, no prior Tab pick) goes to a **routing agent** — a normal `AgentInfo` registry entry whose only job is to classify the user's first message into a target agent name and dispatch — and the router emits a visible dispatch bubble ("→ Reader") so the user sees the routing decision and can correct it next turn if it's wrong. The router is stateless and **first-message-only**: once it picks a downstream agent, that agent owns the rest of the session and the router is never re-consulted; the user has to `/clear` to get back to a router-dispatched first-message. Slash commands and explicit Tab picks bypass the router entirely because they ARE the classification — re-running an LLM call to re-derive the verb the user already typed would be silly. The router is on the agent registry (single model, composes with the existing `compose.ts`/zones/permissions infrastructure) and is included in the Tab cycle alongside Reader/KB so the user can cycle back to "let the router decide" after Tabbing past it; Inkstone treats "an agent whose job is to dispatch" as a real agent in both the architectural and the navigation sense.
