# 0010 — File tools delegated to pi-coding-agent

Inkstone's `read`/`write`/`edit` tools are produced by `@mariozechner/pi-coding-agent`'s factory functions (`createReadTool`/`createWriteTool`/`createEditTool`) called with `VAULT_DIR` as `cwd`; `backend/agent/tools.ts` is a thin wrapper that instantiates them once at module load and exports them for the base layer plus agent `extraTools` to consume. Owning parallel implementations forever was the alternative — pi-coding-agent's versions cover the same vocabulary with meaningfully more rigor (offset/limit reads, image support, multi-edit per call, per-path mutation queueing, truncation hints), and Inkstone wasn't doing anything special there, so the upstream-drift cost is cheaper than maintaining four hand-rolled tool files. The permission dispatcher's path resolver (`permissions.ts:resolvePath`) MUST mirror pi-coding-agent's path-expansion semantics — `~`/`~/` against `$HOME`, leading `@` strip, absolute pass-through, relative against `VAULT_DIR` — otherwise a literal `~/foo` could pass a `startsWith(VAULT_DIR)` sandbox check while the tool writes to `$HOME/foo`; pi-coding-agent doesn't re-export those helpers from its package index, so the subset is inlined. One inert transitive dependency is accepted as part of the trade: pi-coding-agent's tool source imports `@mariozechner/pi-tui` at module scope for unused render hooks that `wrapToolDefinition` strips before handing the tool to pi-agent-core, so pi-tui is module-resolved but never actually executes.
