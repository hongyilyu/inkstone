## Goal

Ship one vertical slice that un-deadens the ADR-0058 media substrate as chat image attachments: (a) a `media/upload` JSON-RPC verb and a `GET /media/{id}` HTTP route on Core; (b) an optional `attachment_ids` array on `run/post_message` and `thread/create` that links uploaded media to the user Message (a `message_parts` row of type 'attachment' + a `media_attachments` row per id); (c) a `Segment::Attachment` variant in the run-timeline union, both languages, with contract fixtures; (d) the composer's disabled Attach chip enabled with file-picker, paste, and drag-drop; (e) inline `<img>` rendering of attached images in the chat feed, live and after reload; (f) base64 `ImageContent` forwarding through the WorkerManifest so vision models see the current turn's images. Boundary: images only, current-turn forwarding only, Message attachment target only; cover images and thumbnails are explicitly cut.

## End state

Core wire surface:
- `media/upload` params `{bytes_base64: String, mime: String, width?: i64, height?: i64}` → result `{media_id: String}`. Handler decodes base64 (invalid → InvalidParams -32602), calls the now-re-exported `db::media::insert_media` with `created_by: "user"`, `created_via_proposal_id: None`, `attachments: Vec::new()` (standalone at upload time; the Message link lands at send).
- `GET /media/{id}` (axum 0.8 `{id}` syntax) → 200 with `Content-Type: <mime>` and the file bytes; 404 for an unknown id or missing file. Wired in main.rs beside the `/ws` route — registered BEFORE `.with_state(state)` since the handler takes `State<AppState>` — using `db::media::get_media` + `db::resolve_media_path`.
- `PostMessageParams` and `ThreadCreateParams` each gain `#[serde(default)] attachment_ids: Vec<String>`. An unknown media id rejects with InvalidParams BEFORE any rows persist. For each valid id, the initial-run transaction writes: one `message_parts` row on the user message (`type='attachment'`, `text=''`, `data` = JSON `{"media_id","mime","width","height"}`) at seq 1..N after the seq-0 prompt part, and one `media_attachments` row (`target_kind='message'`, `target_message_id=<user message>`).
- `crates/core/src/db/media.rs` loses its module-wide `#![allow(dead_code)]` (line 16); `delete_media` alone keeps a fn-level `#[allow(dead_code)]` (still unconsumed). `db/mod.rs` replaces the lines 51-55 "no re-export yet" comment with the real `pub(crate) use media::…` re-export.
- Rust `Segment` gains `Attachment { media_id: String, mime: String, width: Option<i64>, height: Option<i64> }` (optionals `skip_serializing_if`); `db::MessageSegment` (db/mod.rs line 658) gains the mirror variant; `get_thread_with_messages`' user branch emits it from stored attachment parts (one Text segment from concatenated text parts, then one Attachment segment per attachment part, seq order); thread_get.rs maps it 1:1.
- Rust `WorkerManifest` gains `#[serde(skip_serializing_if = "Option::is_none")] attachments: Option<Vec<ManifestAttachment>>` where `ManifestAttachment { mime: String, data_base64: String }` (owned Strings — no lifetime fight with the <'a> struct). `fresh_manifest_line` populates it from the send's attachment ids (read bytes via resolve_media_path, base64-encode); `resume_manifest_line` passes None.

TS / Web:
- packages/protocol: `Segment` union gains the `attachment` variant; `PostMessageParams`/`ThreadCreateParams` gain `attachment_ids: S.optional(S.Array(S.String))`; new `MediaUploadParams`/`MediaUploadResult`; `WorkerManifest` gains optional `attachments`.

## Desired outcome

A user chatting with a vision-capable model can attach one or more images to a message — via the Attach chip, pasting from the clipboard, or dragging files onto the composer — see them render inline in their sent bubble (immediately and after reload), and have the model actually see them. This is the moment Inkstone's chat stops being text-only, and it proves out the pre-paid ADR-0058 media substrate with its first real consumer.

## Out of scope

- Cover images and thumbnails: `media.thumbnail_path` stays unwritten (the migration comment already says so); no image resizing, no dimension extraction Core-side (ADR-0058 scope boundary — the CLIENT may supply width/height at upload, Core never sniffs mime or extracts dimensions).
- No `media/delete` or `media/get` RPC verbs; `db::media::delete_media` stays dead (fn-level allow) — orphan GC remains a non-goal per ADR-0058.
- Attachment targets other than Message (Entity/Observation/Proposal links exist in the substrate but get no wire surface).
- History/resume image replay: only the CURRENT turn's images reach the model; prior-turn attachments are not re-forwarded (`history_for_run` untouched, resume manifest passes None).
- Non-image attachments in the UI (composer accepts image/* only; the wire shapes are mime-general by design, no UI for PDFs etc.).
- No vision-capability gating: if the selected model can't see images the provider's own error surfaces — no catalog-driven disable of the Attach chip.
- No assistant-authored attachments, no attachment support in the faux INKSTONE_FAUX_* worker modes, no Media Entity linkage (ADR-0059 is orthogonal), no upload progress UI beyond the pending-thumbnail state, no WS binary frames (base64 over the existing JSON-RPC text protocol), no configurable size limits.
