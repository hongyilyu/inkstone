# Self-healing decide: typed -32002/-32003 errors, idempotency-keyed retry, already-decided settlement

## Context

Read AGENTS.md (working principles + CI gate) and CONTEXT.md (vocabulary: Proposal, Decision, Parked, Resume) first.

Core already speaks two typed error codes the web client throws away:
- `HandlerError::ProposalNotPending` → `-32002` and `HandlerError::ProviderLoginFailed` → `-32003`, both defined in `crates/core/src/runs/handler.rs:26-77` (code map at lines 55-56; the sanitized message reaches the client — see the existing Rust test `provider_login_failed_frames_minus_32003_with_message` at handler.rs:233).
- `-32002` is framed by `proposal/decide` for lost-race / already-decided / not-parked (`map_decide_error` in `crates/core/src/runs/proposal.rs:90-99`) AND by `proposal/get` when a run has no *pending* proposal (proposal.rs:34-36) — proposal/get is pending-only.
- `-32003` is framed by `provider/login_start` (`crates/core/src/runs/provider.rs:175,227,238` — e.g. "a provider login is already in progress", "provider login failed: <helper message>").

The ui-sdk maps only two codes today: `mapWireError` at `packages/ui-sdk/src/index.ts:182-193` types `-32001 → UnknownThreadError` and `-32602 → InvalidParamsError`; everything else falls through to the untyped `WsRequestError` (which does carry `code`). The `WsError` union is at line 107; `UnknownThreadError` (lines 99-101) is the TaggedError pattern to mirror. `stubWsClient` (lines 329-368) needs no change — no new verbs ship.

Idempotent decide is fully wired server-side but never used by the client: `ProposalDecideParams.decision_idempotency_key` exists in Rust (`crates/core/src/protocol.rs:225-240`, decode tests at 1491-1546) and TS (`packages/protocol/src/index.ts:513-520`), the DB column exists (`crates/core/migrations/0001_initial.sql:124`), and Core's decide precedence (`crates/core/src/decide.rs:175-233`, `compute_outcome`) is: (1) keyed replay — same recorded key returns the prior result, any Run status (decide.rs:197-201); (2) already-decided recovery if the Run is still parked; (3) fresh guarded apply. But `decideProposal` in `apps/web/src/store/bridge.ts:492-545` never sends the key, so a lost-response retry that lands after the Run advanced gets `-32002` and the card wedges in the generic error state.

The second-tab race is real: `send_proposal_changed` is pushed only on the deciding connection's `out_tx` (`crates/core/src/runs/proposal.rs:77,81`; framer in `crates/core/src/runs/reply.rs:82-99`), so a tab that didn't decide never learns the proposal settled — its Accept click returns `-32002` and today lands in the `Exit.isFailure` branch of `decideProposal` (bridge.ts:520-524), which sets proposal status "error" with no useful message.

The already-decided truth IS fetchable: `thread/get` carries a `proposal` segment (`status: accepted|rejected`, `entity_id?`) for the last decided proposal of a run (`crates/core/src/db/mod.rs:838-861`; wire Segment union at `packages/protocol/src/index.ts:284-312`). The web already parses it in `rehydrateDecidedProposals` (`apps/web/src/store/hydrate.ts:107-139`). LANDMINE: the store verb it calls, `rehydrateDecidedProposal` (`apps/web/src/store/chat.ts:322-336`), is skip-if-present — it will NOT update a proposal already in the map (the -32002 case always has one, in `deciding` status). Settlement must go through `setProposalStatus(runId, status, entityId)` (chat.ts:284-310) instead. The seed's "reuse the rehydrateDecidedProposals path" means reuse its *segment-finding logic*, not the store verb.

Settings today: `onConnect` in `apps/web/src/routes/settings/models.tsx:324-336` catches ALL login failures into a boolean `connectFailed` (line 70) rendering fixed copy "Couldn't start the connection. Try Connect again." (lines 375-382). `startLogin` in `apps/web/src/store/providers.ts:30-41` uses `runtime.runPromise`, which wraps failures in Effect's `FiberFailure` — the typed error is unreachable without the `runPromiseExit` + `Cause.squash` pattern established in bridge.ts `send` (lines 248-264) and `useEntityMutation` (`apps/web/src/lib/hooks/useEntityMutation.ts:32-41`).

Decided pill rendering already exists: `ProposalCard` (`apps/web/src/components/ProposalCard.tsx:258-278`) renders the settled inline pill for status accepted/rejected; error state + "Try again" at lines 282, 407-432 (retry re-issues the SAME decision via `lastAttempt`, lines 196-213). Decide flows through `AssistantProposals` (`apps/web/src/components/AssistantProposals.tsx:20-38`) → `decideProposal`.

## Goal

Four client-only slices, Core untouched: (a) add `ProposalNotPendingError` (-32002) and `ProviderLoginFailedError` (-32003) to the ui-sdk typed `WsError` union in `mapWireError`, mirroring `UnknownThreadError`; (b) make `decideProposal` in `apps/web/src/store/bridge.ts` mint one `decision_idempotency_key` per proposal_id (module map, `crypto.randomUUID()`) and send it on every decide/retry of that proposal so a lost-response retry hits Core's keyed replay; (c) on a `ProposalNotPendingError` failure, refetch `thread/get` for the run's thread, find the run's decided `proposal` segment (the same parse `rehydrateDecidedProposals` does), and settle the card via `setProposalStatus` into the existing decided pill — falling back to the current error state if nothing decided is found; do NOT call `proposal/get` (pending-only, itself returns -32002); (d) in Settings → Models, surface the typed -32003 message in the status line instead of the generic connect-failure copy. No new verbs, no protocol/schema changes, no Rust changes.

## End state

- `packages/ui-sdk/src/index.ts`: `ProposalNotPendingError` and `ProviderLoginFailedError` TaggedErrors (each `{ message: string }`) exported; `WsError = WsRequestError | UnknownThreadError | InvalidParamsError | ProposalNotPendingError | ProviderLoginFailedError`; `mapWireError` gains `-32002` and `-32003` arms before the `WsRequestError` fallback.
- `apps/web/src/store/bridge.ts`: a module-level `Map<string, string>` (proposal_id → key) cleared in `resetBridge()`; `decideProposal` passes `decision_idempotency_key` in the `client.proposalDecide({...})` params, minting on first use and reusing on retry. Its failure branch detects `_tag === "ProposalNotPendingError"` (duck-typed off the squashed cause, like `invalidParamsMessage` at bridge.ts:72-84) and runs the settle-from-thread-get path; all other failures keep today's behavior.
- Settle path behavior: `getRunThreadId(runId)` → `client.threadGet(threadId)` → find the message with `run_id === runId` carrying a `proposal` segment whose status narrows to `accepted|rejected` → re-check the currency guard (`getChatState().proposals[runId]` still present — a concurrent cancel clears it) → `setProposalStatus(runId, status, segment.entity_id)` → `interruptRun` + `startRunStream(runtime, threadId, runId)` (mirror of the success path at bridge.ts:537-544, so the other tab's resume tail streams here too). No decided segment found, or the refetch itself fails → `setProposalStatus(runId, "error", ...)` exactly as today.
- The wire shows `{"method":"proposal/decide","params":{...,"decision_idempotency_key":"<uuid>"}}`; two decides of the same proposal carry the same key.
- `apps/web/src/store/providers.ts` `startLogin`: `runPromiseExit` + `Cause.squash` so the rejection carries the real typed error, not a `FiberFailure`.
- `apps/web/src/routes/settings/models.tsx`: `connectFailed: boolean` becomes `connectError: string | null` (or equivalent minimal shape); a `ProviderLoginFailedError` stores its `message` and the status line renders it; any other failure keeps "Couldn't start the connection. Try Connect again."; the `saveStatus` derivation and 2.5s clear timeout keep working.
- User-visible: second tab clicks Accept on an already-decided proposal → card flips to the decided pill ("Added to journal." / "Dismissed."), no error; a lost-response retry settles with Core's prior outcome; Settings shows "provider login failed: …" verbatim.

## Desired outcome

A decide that hits a transient failure or a cross-tab race self-heals instead of wedging. Concretely: (1) if the decide request is lost mid-flight, clicking "Try again" replays the SAME decision against Core's keyed-replay path and settles with the truth instead of double-applying or erroring; (2) if another tab (or an earlier lost-response attempt) already decided the proposal, the review card quietly settles into the normal "Applied."/"Dismissed" pill instead of a scary dead-end error; (3) a failed provider login in Settings shows Core's actual sanitized reason ("a provider login is already in progress", "provider login failed: account locked") instead of one generic line. For a local-first app where the socket drops on every laptop sleep, decide is the one write a user cannot safely spam — this makes it safe.

## Implementation notes

Recommended order — each step ends compiling/green:

1. ui-sdk errors (slice a). In `packages/ui-sdk/src/index.ts`, copy the `UnknownThreadError` pattern (lines 99-101): add `ProposalNotPendingError` and `ProviderLoginFailedError` `Data.TaggedError` classes with `{ message: string }`, extend the `WsError` union (line 107), add two arms to `mapWireError` (lines 182-193). Export both. Grep for exhaustive `WsError` matches first — there are none today (consumers duck-type `_tag` — bridge.ts:72-84 — or `Effect.catchTag` specific tags — hydrate.ts:199-204), so widening is safe; `connectionFailureCopy` (`apps/web/src/lib/connectionFailureCopy.ts:41-55`) only matches `WsRequestError` and is unaffected.

2. Idempotency key (slice b). In `apps/web/src/store/bridge.ts`: module map `const decisionKeys = new Map<string, string>()` beside `fibers`; in `decideProposal`, `const key = decisionKeys.get(proposal.proposal_id) ?? crypto.randomUUID(); decisionKeys.set(proposal.proposal_id, key);` and add `decision_idempotency_key: key` to the `client.proposalDecide` params (lines 511-516). Key per proposal_id, NOT per attempt — Core's keyed replay (decide.rs:197-201) means a retried decide with the same key returns the prior result, which is exactly what "Try again" (ProposalCard `retry`, re-issuing the same decision) needs. Clear the map in `resetBridge()` (bridge.ts:64-68) for test isolation. The TS `ProposalDecideParams` schema already accepts the field — no protocol edit.

3. -32002 settlement (slice c). Extract the decided-segment finder from `rehydrateDecidedProposals` (`apps/web/src/store/hydrate.ts:107-139`) into a small exported pure helper, e.g. `decidedProposalSegment(views: ThreadGetResult["messages"], runId: string): { status: "accepted" | "rejected"; entity_id?: string } | undefined` — keep the open-string narrowing (`status !== "accepted" && status !== "rejected" → skip`, hydrate.ts:122-125) — and have `rehydrateDecidedProposals` keep working as-is (it iterates all views; the helper can serve just the bridge, or refactor minimally — do not gold-plate). In `decideProposal`'s failure branch (bridge.ts:520-524), branch on the squashed error's `_tag === "ProposalNotPendingError"` (duck-type like `invalidParamsMessage`, bridge.ts:72-84): fetch `client.threadGet(getRunThreadId(runId))` via `runtime.runPromiseExit`, find the segment, then settle. LANDMINES: (i) `rehydrateDecidedProposal` (chat.ts:322) is skip-if-present and will silently no-op on the existing `deciding` record — use `setProposalStatus(runId, status, entityId)`; (ii) re-check `getChatState().proposals[runId] !== undefined` AFTER the await (the currency guard at bridge.ts:531 exists because `cancelRun` can clear the proposal mid-flight); (iii) `getRunThreadId` (chat.ts:250) works here because a parked run has a live RunRecord — but guard `undefined` anyway; (iv) after settling, mirror the success path (bridge.ts:537-544): `interruptRun` + `startRunStream` so the resumed run's tail streams into this tab; (v) `-32002` also fires for "run not parked while proposal still pending" — then no decided segment exists in thread/get and you must fall back to `setProposalStatus(runId, "error", undefined, ...)`; do NOT loop or refetch `proposal/get`.

4. -32003 in Settings (slice d). First fix `startLogin` (`apps/web/src/store/providers.ts:30-41`) to `runPromiseExit` + `Cause.squash` + `throw` (copy `useEntityMutation`'s mutationFn, `apps/web/src/lib/hooks/useEntityMutation.ts:32-41`) — without this the catch in models.tsx sees a `FiberFailure`, not the tagged error. Then in `apps/web/src/routes/settings/models.tsx`: replace `connectFailed` boolean (line 70) with `connectError: string | null`; in `onConnect`'s `.catch` (line 332, inside the 324-336 callback), duck-type `_tag === "ProviderLoginFailedError"` and a non-empty `message` → store it, else store the generic copy; render `connectError` in the status line (replacing the fixed string at lines 379-381); keep `saveStatus` error-precedence (line 110) and the timeout clear (lines 210-219, `setConnectFailed(false)` → `setConnectError(null)`).

Commits (AGENTS.md format), one per slice or grouped sensibly: `feat(ui-sdk): type -32002/-32003 in mapWireError`, `feat(web): idempotency-keyed decide + already-decided settlement`, `feat(web): surface typed provider-login failure in settings`.

## Out of scope

- Any Rust/Core change: the codes, keyed replay, DB column, and notifications all exist. No fan-out of `proposal/changed` to all connections (the principled fix for the second-tab race — a different, larger feature).
- No `-32004` rework (`ProviderNotConnected` stays duck-typed on `WsRequestError.code` in connectionFailureCopy.ts).
- No protocol/schema or contract-fixture changes (`decision_idempotency_key` already exists on both sides).
- No persistence of idempotency keys across page reloads (a reload mints fresh keys; Core's precedence step 2 — already-decided recovery while parked — covers the reload-then-retry case).
- No polling/auto-retry loops; settlement is one refetch on the typed error.
- No changes to `IntentGraphReviewCard`'s internal decision UI beyond what flows through the shared `decideProposal` (it already routes through the same bridge verb).
- No stubWsClient changes, no new SDK verbs.
- Don't rewrite `connectFailed`'s timeout/ack machinery in models.tsx — minimal state widening only.
